## Решения по архитектуре
- 2026-07-09: **Сессия играет ОДИН эпизод; список эпизодов — `@export` на роутере (`game.tscn`), не на сессии.** Уточнение плана «`Main.episodes: Array[Episode]`»: экран выбора эпизода строит роутер ДО создания сессии, а конец эпизода всё равно возвращает в главное меню (сессия сносится) — значит, сессии весь список не нужен, ей передаётся один выбранный `Episode` (`set()` до `add_child`, перекрывает отладочный эпизод из инспектора `main.tscn`). Бонус: сброс лоадаута между эпизодами получается БЕСПЛАТНО (новая сессия = свежий игрок с `@export`-дефолтами), кода сброса нет вовсе. Пара (эпизод, уровень) для облачного резюма Этапа 4: эпизод знает роутер, уровень — `set_start_level` сессии.
- 2026-07-09: **Раскладка уровней — папка на эпизод + схема имён `level_E<эпизод>_L<уровень>`** (решение Сергея). Уровни кампании лежат в `scenes/levels/episode_<N>/` (папки snake_case — как `pickups/{ammo,health,armor}` и прочие, решение 2026-06-18), имя файла `level_E<N>_L<уровень>` дублирует эпизод+уровень, чтобы файл читался при глобальном поиске без пути (`level_E2_L1` = эпизод 2, уровень 1). Прежняя плоская сквозная `level_doom_0N` структуру не отражала. Первый уровень (`episode_1/level_E1_L1`) собирает `build_doom_kit.gd` (`LEVEL_PATH`/`root.name` под схему); остальные — `gen_demo_levels.py` (кладёт в `episode_<N>/` по цифре из node-name `LevelE1L2`). Пути ВНУТРИ `.tscn` уровней к киту/скриптам — абсолютные `res://…`, поэтому перемещение файла глубже их не ломает. Переименование/перемещение сделано вне редактора Godot (`git mv` + правка всех `path=`/`uid=` в ссылающихся `game.tscn`/`main.tscn`): для `.tscn` без `.import`-сайдкаров это безопасно, т.к. Godot резолвит `ext_resource` в первую очередь по `uid` (у файлов он не менялся, кроме говорящих uid генератов, обновлённых согласованно и в файле, и в `game.tscn`). Проверено автоскриптом: все ссылки/uid/`load_steps` сходятся, плоских/старых путей не осталось. `test_level`/`test_level_2` в схему не входят (отладочные, в корне `scenes/levels/`).

Иной тип атаки (ближний бой без луча, ракета-снаряд с AoE) = наследник с переопределённым `_fire`, ядро не трогаем. `fire_mode` (`SEMI`/`AUTO`) — данные; авто-логику держит менеджер (повтор `try_fire`, темп задаёт кулдаун).

Поэтому дымок — пул переиспользуемых CPU-частиц (идут на всех рендерах) с биллбордом и сгенерированной кодом текстурой (ноль веса). Связь — signal up: `Weapon.surface_hit(position, normal)` (только по неуязвимым поверхностям) → `WeaponManager` пробрасывает → `main` зовёт `ImpactSmoke.spawn`. Если Safari заартефачит сортировку прозрачности дыма — перейти на `BLEND_MODE_ADD`/альфа-скиссор.

Оружие не владеет патронами: объявляет `ammo_type` + `ammo_per_shot` и тянет из пула. Несколько стволов могут делить один пул (как пистолет+пулемёт в DOOM). Новый снаряд = новый `ammo_type` + свой пул, ядро не трогаем. Состояние — в компоненте, не в autoload-`GameState` (как и здоровье).

- 2026-06-17: **Конфиг пулов — ресурс `AmmoType`** (`@export id/max/start`), массив в инспекторе `AmmoComponent`. Типобезопасно и расширяемо (добавить тип = элемент массива), вместо параллельных `Dictionary` с риском рассинхрона ключей.
- 2026-06-17: **`PickupBase` — база + хук `_apply`** (как `EnemyBase`). `Area3D`, реагирует на группу `target_group`, единоразовый подбор. Эффект задаёт наследник (`_apply(body) -> bool`): true — пикап исчез, false — не нужен, остался лежать (DOOM-овское «не подобрать, если полно»). HUD — чистое отображение (`set_health`, `show_game_over`) + сигнал `restart_requested` наверх; об игроке и `main` он не знает. `HealthComponent` игрока эмитит `health_changed`/`died`, потребители (HUD, `main`) подписываются сами (signal up).
