Обновлён: 2026-06-18
Движок: Godot 4.6.3 (GDScript) · Подход: low-poly 3D со стилизацией под ретро · Платформа: Яндекс Игры (HTML5)

## Сейчас в работе

Перетряхнули папки (`assets/`, разбивка `scripts/`, пикапы по категориям), удалили мёртвый ассет, расцепили `_setup_ammo`, добавили `@tool`-превью спрайта врага в редакторе. Детали — в «Сделано».

Захватывающий клик поглощается в `player._input`, выстрел — только при захваченном курсоре. Звук — опциональный `@export shot_sound: AudioStream` (плейсхолдер молчит, пока ассет не назначен). Скрипты: `res://scenes/player/weapon.gd`, `res://scenes/ui/crosshair.gd`; правка `res://scenes/player/player.gd` (захват курсора перенесён в `_input`).

(Дымок от попадания по стене — сделано через `ImpactSmoke`. Полноценные декали-дырки на стенах не делаем: узел `Decal` не поддержан рендером Compatibility/WebGL.)

## Решения по архитектуре
- 2026-06-18: **Структура папок — разбивка по роли, ассеты в `assets/`.** `scripts/` дробится на `components/resources/effects/sprites/ui` (плоская папка начинала мешать сорта скриптов); пикапы — по категориям `pickups/{ammo,health,armor}/` с общей базой `pickup_base` в корне; ассеты — в `assets/sprites/`. Конвенция прежняя: скрипт со своей сценой лежит рядом со сценой, скрипт без сцены — в `scripts/` (потому `crosshair` уехал туда из `scenes/ui/`). Перемещения делать ТОЛЬКО внутри редактора Godot — он переписывает `uid`/`path`/`.import`; вне редактора ломает ссылки.
- 2026-06-18: **Редакторное превью спрайта — `@tool` + `Engine.is_editor_hint()`, без правок сцен.** `DirectionalSprite3D` в редакторе сам раскладывает атлас (читает `columns`+`animations`) и показывает один idle-кадр; рантайм-логика (анимация, выбор ракурса по камере) заглушена. Альтернативу «вручную `hframes`/`vframes`/`frame` в каждой сцене врага» отвергли — `@tool` делает это автоматически для любого врага и не плодит магических чисел в сценах. Кадр-превью задаётся `@export editor_preview_animation`.
- 2026-06-18: **Броня — отдельный компонент `ArmorComponent` перед HP, не правка `HealthComponent`.** Модель DOOM 1/2: поглощает долю урона по классу (зелёная 1/3, синяя 1/2), пул списывается 1:1, на нуле урон идёт целиком в HP. Игрок маршрутизирует `take_damage` через броню → HP; источник урона по-прежнему знает только `take_damage`. Доли — `@export absorption_by_class` (данные, не код).
- 2026-06-18: **Оверхил — в `HealthComponent` аддитивно, без форка.** `overheal_max` (потолок поверх обычного `max_health`) + `overheal_decay` (тает обратно к базе в `_process`). Дефолты 0 → у врагов оверхила и таяния нет, поведение не изменилось. Оверхил дают только «оверхильные» предметы (флаг `allow_overheal` у пикапа: бонус-склянка, сфера души).
- 2026-06-18: **Предметы — префаб на предмет (сцена-наследник категорийной базы), не override в инспекторе при расстановке.** Базы `health_pickup.tscn`/`armor_pickup.tscn` несут скрипт; конкретные предметы — наследованные сцены с запечёнными `@export` (+цвет-плейсхолдер). Расстановка = перетащить префаб. Цифры/режим — данные на префабе, новый предмет = ещё один наследник, кода нет.
- 2026-06-17: **Эффекты-всплески частиц — общая база `ParticleBurst3D` + наследники.** Пул `CPUParticles3D` (round-robin), общий меш/материал и `spawn(position, normal)` живут в базе; конкретный эффект задаёт текстуру/настройки через хуки (`_make_texture`/`_configure_emitter`/`_quad_size`). `ImpactSmoke` (дым по стенам) и `HitStars` (звёзды по врагу) — тонкие наследники, пул не копипастится. Третий эффект = ещё один наследник.
- 2026-06-17: **Попадание по врагу — отдельный сигнал `damageable_hit`, параллельный `surface_hit`.** В `_fire` это разные ветки (есть `take_damage` → враг; иначе → поверхность). Каждая шлёт свой сигнал с `(position, normal)`; `WeaponManager` пробрасывает оба наверх; `main` вешает дым на один, звёзды на другой. Эффект не зашит в оружие — подписчик решает.
- 2026-06-17: **Реакция на урон у врага — спрайт боли, не `modulate`.** Анимация `pain` + `hurt()` + `pain_hold` с «замком»: пока играет боль, `play()` (ходьба из чейза каждый кадр) её не перебивает, только запоминает желаемое; по таймеру — возврат. `play(name, force=true)` для смерти перебивает боль. Цветовая вспышка `modulate` убрана как избыточная.
- 2026-06-17: **Анимация спрайта — скрипт на самом `Sprite3D` (`DirectionalSprite3D`), не отдельный узел-компонент.** Спрайт сам себя анимирует: реюзабелен (тот же скрипт на любом враге), один атлас на врага, `flip_h`/`frame` тривиально. `hframes/vframes/frame` вместо `AnimatedSprite3D` — легче и полный контроль выбора ракурса. Конфиг анимаций — массив ресурсов `SpriteAnimation` (как `AmmoType`), новый враг = атлас + массив, кода нет.
- 2026-06-17: **«Взгляд» врага = поворот тела (`look_at` по направлению движения/цели).** Биллборд Y визуально тело не вращает, капсула симметрична — поворот нужен лишь как опорное направление для выбора ракурса. Ходьба смотрит по `velocity` (отсюда видны бока при обходе игроком), атака — на цель.
- 2026-06-17: **Смерть — ненаправленная анимация (один ракурс, как в DOOM).** `directional=false` → всегда столбец 0. Кадры смерти кладутся в столбец 0 строк 6–9, остальные столбцы этих строк пустые; сетка остаётся единой `5×N`.
- 2026-06-17: **Атлас пиксель-арта импортируется Lossless, без мипмапов, без авто-VRAM** (`detect_3d/compress_to=0`). VRAM-сжатие (S3TC/ETC) мылит низкий пиксель-арт. Низкое разрешение — главный рычаг веса: атлас всех ракурсов+кадров (21 КБ) легче одного старого кадра (2.1 МБ).
- 2026-06-17: **Вспышка боли — `Tween` на `modulate` спрайта**, по `HealthComponent.health_changed` (signal up), кроме добивающего удара (там играет смерть). Кадр под «боль» не рисуем — это цветовая вспышка поверх текущего кадра.
- 2026-06-17: **Несколько оружий — менеджер слотов + база-ствол (данные, не классы).** `WeaponManager` владеет дочерними `Weapon`, переключает активный (видимость + роутинг ввода). `Weapon` — одна база, конфигурируется через `@export` (урон/режим/кулдаун/дробины/разброс/тип патронов/слот); новый ствол = сцена-дубликат с другими цифрами, копипасты классов нет. Слот объявляет сам ствол (`@export slot`), менеджер авто-раскладывает детей по слотам в `_ready`.
- 2026-06-17: **Что делает выстрел — хук `_fire()` (база+хук, как `EnemyBase`/`PickupBase`).** Дефолт — hitscan (веер `pellets` лучей с разбросом). Иной тип атаки (ближний бой без луча, ракета-снаряд с AoE) = наследник с переопределённым `_fire`, ядро не трогаем. `fire_mode` (`SEMI`/`AUTO`) — данные; авто-логику держит менеджер (повтор `try_fire`, темп задаёт кулдаун).
- 2026-06-17: **Роли слотов и общие пулы (модель DOOM 1/2):** 1 — ближний бой (нет патронов, позже), 2 — пистолет (`bullets`), 3 — дробовик (`shells`), 4 — пулемёт (`bullets`, делит пул с пистолетом), 5 — ракетомёт (`rockets`, позже). Несколько стволов делят один пул через `AmmoComponent` без правок ядра.
- 2026-06-17: **Переключение оружия:** клавиши `weapon_1..5` — прямой выбор слота, колёсико (`weapon_next`/`weapon_prev`) — листание по занятым. Захватывающий клик по-прежнему не стреляет (поглощается `player._input`), поэтому первый клик не запускает авто-огонь по ошибке.
- 2026-06-17: **Эффект попадания — `CPUParticles3D`, не узел `Decal`.** Узел `Decal` не работает в рендере Compatibility (WebGL) — проверено по докам/issue, ограничение остаётся и в 4.6. Поэтому дымок — пул переиспользуемых CPU-частиц (идут на всех рендерах) с биллбордом и сгенерированной кодом текстурой (ноль веса). Связь — signal up: `Weapon.surface_hit(position, normal)` (только по неуязвимым поверхностям) → `WeaponManager` пробрасывает → `main` зовёт `ImpactSmoke.spawn`. Если Safari заартефачит сортировку прозрачности дыма — перейти на `BLEND_MODE_ADD`/альфа-скиссор.
- 2026-06-17: **Патроны — типизированные пулы на игроке, не на оружии (модель DOOM 1/2).** Боезапас живёт в `AmmoComponent` на игроке (сосед `HealthComponent`), ключ — тип патрона (`StringName`). Оружие не владеет патронами: объявляет `ammo_type` + `ammo_per_shot` и тянет из пула. Несколько стволов могут делить один пул (как пистолет+пулемёт в DOOM). Новый снаряд = новый `ammo_type` + свой пул, ядро не трогаем. Состояние — в компоненте, не в autoload-`GameState` (как и здоровье).
- 2026-06-17: **Расход — классика DOOM 1/2: один пул, без перезарядки.** Выстрел уменьшает пул, на нуле — сухой щелчок. Обойма/reload при желании — надстройка над тем же пулом позже, фундамент не ломает.
- 2026-06-17: **Конфиг пулов — ресурс `AmmoType`** (`@export id/max/start`), массив в инспекторе `AmmoComponent`. Типобезопасно и расширяемо (добавить тип = элемент массива), вместо параллельных `Dictionary` с риском рассинхрона ключей.
- 2026-06-17: **`PickupBase` — база + хук `_apply`** (как `EnemyBase`). `Area3D`, реагирует на группу `target_group`, единоразовый подбор. Эффект задаёт наследник (`_apply(body) -> bool`): true — пикап исчез, false — не нужен, остался лежать (DOOM-овское «не подобрать, если полно»). Ось расширения пикапов — поведение (свой скрипт-наследник, как `AmmoPickup`), а не «цифры одной сцены». Аптечка/броня — следующие наследники (аптечке нужен `heal` на `HealthComponent`).
- 2026-06-17: **Конвенция пополнения — `add_ammo` на теле игрока**, делегирует в `AmmoComponent` (зеркало `take_damage`→`HealthComponent`). Пикап (внешний источник) не знает о внутренностях игрока: `has_method("add_ammo")` → вызов. Оружие (внутренняя часть игрока) обращается к `AmmoComponent` напрямую (нашло на теле в `_ready`).
- 2026-06-17: **HUD показывает патроны активного типа.** `ammo_changed` несёт `type`; HUD фильтрует по `_active_type` (его задаёт `main` по активному оружию; при переключении стволов — активный ствол). Неактивные пулы на HUD не мелькают. Сейчас тип один.
- 2026-06-17: **HUD и game-over живут в `main`.** Точка входа инстансит HUD, связывает его со здоровьем игрока и владеет паузой/рестартом. HUD — чистое отображение (`set_health`, `show_game_over`) + сигнал `restart_requested` наверх; об игроке и `main` он не знает. `HealthComponent` игрока эмитит `health_changed`/`died`, потребители (HUD, `main`) подписываются сами (signal up).
- 2026-06-17: **Смерть = пауза всей игры + оверлей, рестарт по клавише.** На `died`: `get_tree().paused = true` (враги/игрок/оружие замирают), мышь в `VISIBLE`, красный оверлей «YOU DIED». HUD в `PROCESS_MODE_ALWAYS`, чтобы ловить ввод на паузе; по любой клавише/клику — `restart_requested` → `main` снимает паузу и `reload_current_scene()`. Перезагрузка всей сцены — простой срез; умный респаун без перезагрузки — позже, при необходимости.
- 2026-06-17: **HUD рисуется кодом** (`_draw`, дефолтный шрифт), как крестик/ствол — ноль веса билда. Патроны пока «∞»: системы патронов нет (оружие — бесконечный hitscan), её вводим на Этапе 2 вместе с пикапами/несколькими стволами — тогда «∞» заменится реальным счётчиком. Пиксель-шрифт/спрайты HUD — позже, в полировку.
- 2026-06-17: **Грабля (на будущее):** после `reload_current_scene()` / смены сцены нельзя обращаться к `self` или его `get_viewport()` — узел уже выкинут из дерева (`null`). `emit()` сигнала рестарта выполняется синхронно и сносит узел, поэтому `set_input_as_handled()` зовём ДО `emit()`, а сам `emit()` — последним действием.
- 2026-06-17: **Конвенция урона** — у всех damageable-объектов метод `take_damage(amount: float)`. Источник урона (оружие/враг) не знает о классах целей: проверяет `has_method("take_damage")` и вызывает. Метод лежит на физическом теле (по нему попадает луч/контакт) и делегирует во вложенный `HealthComponent`.
- 2026-06-17: **`HealthComponent` универсален для врага и игрока.** Компонент знает только числа (`current/max`, кламп, `take_damage`, сигналы `health_changed`/`died`). Что значит «умереть»/«получить урон» — решает владелец, подписавшись на сигналы (signal up): враг → труп+`queue_free`, игрок → game over/рестарт+HUD. Броня, i-frames, оверхил, `heal` — добавляются аддитивно (опц. `@export` или отдельное звено перед HP), форка компонента не требуют. Файл: `res://scripts/health_component.gd`.
- 2026-06-17: **Расширяемость врагов — база + хуки + статы.** `EnemyBase` (`class_name`, `extends CharacterBody3D`) = готовый наземный враг ближнего боя и одновременно база. Новый тип врага: (1) другие цифры — унаследованная сцена со своими `@export`, ноль кода; (2) другое поведение — `extends EnemyBase` с переопределением хуков (`_move_towards_target` — летающий; `_perform_attack` — стреляющий; `_can_see_target` — конус/360°; `_on_death` — особая смерть). Типы врагов оформляем унаследованными сценами от `enemy_base.tscn`.
- 2026-06-17: **Враг — спрайт-биллборд + отдельный хитбокс.** Визуал `Sprite3D` (`Billboard = Y-Billboard`, `Texture Filter = Nearest`, `Alpha Cut = Discard` — жёсткие края и обход сортировки прозрачности WebGL2/Safari, `Pixel Size ≈ 0.04`). Коллизия — отдельный `CollisionShape3D` (капсула) на теле; по спрайту луч не ловится, стреляем по телу (слой 1, как стены — текущий hitscan попадает без правок оружия). Плейсхолдер-текстура генерируется кодом (`Image`/`ImageTexture`, ноль веса билда); появится арт — назначить `texture`, генерация сама отключится.
- 2026-06-17: **Обнаружение врага — дистанция + линия видимости.** В `detection_range` И прямой луч (`intersect_ray` с высоты «глаз», исключая себя) упёрся в саму цель или ни во что — иначе не «видит» сквозь стены.
